# 架构决策记录 (ADR)

## ADR-001: 采用 Harness Engineering 三分类架构

**状态**: 已采纳

**背景**: 需要一种系统化方法来管理 AI 驱动的开发流程，确保代码质量和可维护性。

**决策**: 采用 Martin Fowler 提出的 Harness Engineering 三分类架构：
1. Context Engineering - 上下文管理
2. Architectural Constraints - 架构约束
3. Garbage Collection - 垃圾回收

**理由**:
- 清晰关注点分离
- 自动化质量保障
- 可扩展的知识管理

**后果**:
- 需要维护完整的文档结构
- 需要配置 linter 和分层检查
- 需要实现园丁代理

## ADR-002: 可观测性栈选型

**状态**: 已采纳

**背景**: 需要选择可观测性工具栈来监控 AI 系统行为。

**决策**: 采用以下技术栈：
- **指标**: Prometheus + Grafana
- **追踪**: OpenTelemetry
- **日志**: Loki

**理由**:
- 开源且社区活跃
- 组件间集成良好
- 云原生标准

## ADR-003: 园丁代理设计

**状态**: 已采纳

**背景**: 需要自动化管理代码库健康度。

**决策**: 实现园丁代理，负责：
- 代码过期检测
- 依赖更新
- 死代码清理
- 文档同步

**理由**:
- 减少人工维护负担
- 保持代码库健康
- 自动化技术债务管理
