### 2.3 MVP 核心功能（已调整）

### 2.4 关键设计原则（已调整）

| 原则 | 说明 |
|------|------|
| **PC桌面应用** | 本地运行，数据不出机器 |
| **科技风界面** | 深色主题，精致表格，代码高亮 |
| **类微信交互** | 左侧联系人，右侧聊天，降低学习成本 |
| **开箱即用** | 预配置专家，不写提示词 |
| **并行效率** | 多专家同时响应，一次获得多角度反馈 |
| **透明决策** | 冲突时展示各方观点，不替用户做决定 |
| **ROI 可追踪** | Dashboard 清晰展示时间和成本节省 |

## 三、关键风险与假设（已更新）

### 3.1 关键假设

1. 开发者愿意为节省时间付费（ROI 驱动）
2. 并行调用多个 LLM 的成本可接受
3. 预配置专家能满足 80% 场景需求
4. PC 桌面应用比 Web 更受欢迎

### 3.2 主要风险

| 风险 | 影响 | 缓解策略 |
|------|------|----------|
| 并行 LLM 调用成本高 | ROI 打折扣 | 智能缓存 + 本地模型选项 |
| 预配置专家不够精准 | 用户需要频繁自定义 | 丰富的模板 + AI 生成 |
| 首次调用等待时间长 | 用户流失 | 显示进度 + 预加载 |
| Electron 应用体积大 | 下载安装门槛 | 提供轻量版 CLI |

## 五、访谈关键洞察

### 用户原话

> "我希望是多专家同时参与，用户只要提出一个想法，多专家可以在自己的领域协同工作"

> "如果有现成的专家配置，最好不要用户折腾，用户不喜欢麻烦"

> "界面应该类似微信，左侧是联系人列表，右侧是聊天记录"

> "我希望是一个终端，不做成网页，它应该是一个单独的应用"

> "效率指标，进度指标，成本指标都要有，dashboard应该是一系列精美的表格组成"

### 关键决策

| 决策 | 依据 |
|------|------|
| PC桌面应用（非 Web） | 用户明确需求，科技风界面 |
| 本地运行（非云端） | 保护代码安全，数据不出机器 |
| 多专家并行（非串行） | 核心价值，3x 效率提升 |
| 冲突时列出观点（非自动仲裁） | 用户要求透明决策 |
| Dashboard 追踪 ROI | 用户要求证明价值 |
| 预配置专家（非自定义） | 用户不喜欢折腾配置 |
