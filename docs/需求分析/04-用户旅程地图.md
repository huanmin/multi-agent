## 旅程 1: 开发者从发现到深度使用

### 旅程概览

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  用户: 全栈开发者                                                            │
│  场景: 使用平台辅助软件开发全流程                                            │
│  时长: 第 1 天到第 30 天                                                     │
└─────────────────────────────────────────────────────────────────────────────┘

情绪曲线:
📈       📈📈       📉📈       📈📈       📈📈       📈📈
  \     /    \     /    \     /    \     /    \     /
   \   /      \   /      \   /      \   /      \   /
    \ /        \ /        \ /        \ /        \ /
     📉        📉📉        📉        📉        📉

    下载      首次使用   多专家并行  日常使用   Dashboard  深度定制
    安装        体验        体验      阶段       追踪        阶段
```

---

### 阶段 1: 下载与安装 (Day 0)

| 维度 | 内容 |
|------|------|
| **目标** | 快速下载并安装应用 |
| **触点** | 官网 → 下载页 → 安装包 → 安装向导 |
| **行为** | ① 在官网看到产品介绍<br>② 下载对应系统版本（macOS/Windows/Linux）<br>③ 运行安装程序<br>④ 完成安装 |
| **想法** | "界面看起来很酷，科技风"<br>"安装包有点大，希望值得"<br>"安装还挺快的" |
| **情绪** | 📈 好奇 → 📉 等待 → 📈 顺利 |
| **痛点** | ① 安装包体积大，下载慢<br>② 不确定是否支持自己的系统版本 |
| **机会** | ✅ 提供轻量版下载选项<br>✅ 明确系统要求和兼容性 |

---

### 阶段 2: 首次使用 (Day 1)

| 维度 | 内容 |
|------|------|
| **目标** | 快速体验核心价值 |
| **触点** | 欢迎页 → 预配置工作组 → 群聊界面 → 首次对话 |
| **行为** | ① 打开应用，看到欢迎引导<br>② 浏览预配置工作组（技术方案设计、代码审查等）<br>③ 选择「技术方案设计」组<br>④ 看到群组已有：产品、架构、前后端、安全、性能专家<br>⑤ 输入："设计一个用户积分系统"<br>⑥ 多个专家同时开始响应 |
| **想法** | "开箱即用，不用配置，爽！"<br>"哇，多个专家同时在回答！"<br>"架构师说用 Redis，性能专家说用本地缓存，这就是冲突检测？" |
| **情绪** | 📈 惊喜 → 📈 震撼 → 📈 满意 |
| **痛点** | ① 首次调用等待时间较长<br>② 不清楚并行调用是否消耗更多 token |
| **机会** | ✅ 显示预计等待时间和 token 消耗估算<br>✅ 首次使用引导提示 |

---

### 阶段 3: 多专家并行体验 (Day 1-3)

| 维度 | 内容 |
|------|------|
| **目标** | 深入体验多专家协作的价值 |
| **触点** | 群聊界面 → @提及 → 追问 → 冲突解决 |
| **行为** | ① 查看各专家发言，对比不同角度<br>② 发现架构师和性能专家意见冲突<br>③ 查看冲突对比视图<br>④ 决策选择 Redis 方案<br>⑤ @架构师追问 "Redis 数据结构怎么设计？"<br>⑥ 继续多轮讨论，完善方案 |
| **想法** | "一次获得多角度反馈，太高效了！"<br>"冲突对比很清晰，优缺点一目了然"<br>"@功能很精准，不会打扰其他专家" |
| **情绪** | 📈 高效 → 📈 掌控 → 📈 依赖 |
| **痛点** | ① 追问时上下文衔接有时不连贯<br>② 专家发言太多，需要滚动查看 |
| **机会** | ✅ 优化上下文管理<br>✅ 消息折叠/分组功能 |

---

### 阶段 4: 日常使用 (Day 4-14)

| 维度 | 内容 |
|------|------|
| **目标** | 将平台融入日常开发工作流 |
| **触点** | 固定工作组 → 代码审查 → 导出文档 |
| **行为** | ① 每天打开应用，进入固定工作组<br>② 遇到技术问题，描述需求获得多专家反馈<br>③ 写代码前先用代码审查组预审查<br>④ 导出技术方案为 Markdown，存到项目文档<br>⑤ 创建多个固定组（技术方案、代码审查、Bug分析等） |
| **想法** | "已经离不开了，效率提升明显"<br>"代码审查帮我发现了很多低级错误"<br>"导出功能很方便，直接存文档" |
| **情绪** | 📈 高效 → 📈 满意 → 📈 习惯 |
| **痛点** | ① 希望能在 IDE 里直接使用<br>② 想看到具体节省了多少时间 |
| **机会** | ✅ 提示 VS Code 插件（V1.2）<br>✅ Dashboard 功能（V1.1） |

---

### 阶段 5: Dashboard 追踪 (Day 15-21)

| 维度 | 内容 |
|------|------|
| **目标** | 查看效率提升和成本节省 |
| **触点** | Dashboard 入口 → 效率指标 → 成本指标 → 进度指标 |
| **行为** | ① 切换到 Dashboard 视图<br>② 查看本周数据：完成 25 个任务，节省约 10 小时<br>③ 查看成本：消耗 token 价值 ¥30，相当于节省人工成本 ¥500<br>④ 查看项目进度：项目 A 完成 80%<br>⑤ 导出周报 |
| **想法** | "ROI 300%，值！"<br>"原来这周做了这么多事"<br>"可以向上级证明工具的价值了" |
| **情绪** | 📈 惊喜 → 📈 满足 → 📈 推荐意愿 |
| **痛点** | ① 时间节省是估算，不够精准<br>② 想和上周对比趋势 |
| **机会** | ✅ 让用户标记任务实际耗时，校准估算<br>✅ 趋势图和对比功能 |

---

### 阶段 6: 深度定制 (Day 22-30)

| 维度 | 内容 |
|------|------|
| **目标** | 自定义专家和配置 |
| **触点** | 专家市场 → AI 生成专家 → Git 同步 → 插件开发 |
| **行为** | ① 从专家市场下载 "微服务专家"<br>② 让 AI 生成一个 "区块链专家"<br>③ 调整专家提示词，保存自定义版本<br>④ 用 Git 管理专家配置<br>⑤ 分享给同事（各自独立使用） |
| **想法** | "自定义专家很灵活"<br>"Git 管理配置，很程序员"<br>"同事也能用我的配置，不错" |
| **情绪** | 📈 掌控 → 📈 归属 → 📈 影响力 |
| **痛点** | ① AI 生成的专家有时不够精准<br>② 想开发自己的工具插件 |
| **机会** | ✅ 优化 AI 生成质量<br>✅ 开放插件 SDK（V1.2） |

## 旅程 2: 代码审查专项旅程

### 旅程概览

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  用户: 开发者（代码提交前）                                                  │
│  场景: 使用平台做代码审查                                                    │
│  频率: 每次提交代码前                                                        │
└─────────────────────────────────────────────────────────────────────────────┘

情绪曲线:
     📉       📈📈       📈📈
       \     /    \     /
        \   /      \   /
         \ /        \ /
          📈        📉

       提交代码    获得反馈    修复问题    审查通过
       前焦虑      多维度       明确       放心提交
```

---

### 阶段 1: 提交前焦虑

| 维度 | 内容 |
|------|------|
| **目标** | 确保代码质量，避免低级错误 |
| **触点** | IDE → 代码审查群组 → 代码输入 |
| **行为** | ① 完成一段代码编写<br>② 感觉有些不放心<br>③ 打开平台的「代码审查」群组<br>④ 粘贴代码 |
| **想法** | "这段逻辑有点复杂，让专家看看"<br>"希望别有什么低级错误"<br>"比让同事审查快多了" |
| **情绪** | 📉 焦虑 → 📈 期待 |

---

### 阶段 2: 获得多维度反馈

| 维度 | 内容 |
|------|------|
| **目标** | 快速获得审查意见 |
| **触点** | 代码审查群组 → 各专家审查结果 |
| **行为** | ① 代码规范专家指出命名问题<br>② 性能专家指出缺少索引<br>③ 安全专家提示输入未校验<br>④ 查看行级评论 |
| **想法** | "幸亏发现了，差点就提交了"<br>"三个维度同时审查，全面"<br>"这个 N+1 问题我自己都没注意到" |
| **情绪** | 📈 惊喜 → 📈 感激 |

---

### 阶段 3: 修复问题

| 维度 | 内容 |
|------|------|
| **目标** | 根据审查意见修复代码 |
| **触点** | 审查结果 → IDE → 重新审查 |
| **行为** | ① 在 IDE 中修改代码<br>② 回到平台标记问题为已修复<br>③ 重新提交审查<br>④ 确认通过 |
| **想法** | "修复后重新审查很方便"<br>"可以过滤只看未修复的问题" |
| **情绪** | 📉 压力 → 📈 成就感 |

---

### 阶段 4: 放心提交

| 维度 | 内容 |
|------|------|
| **目标** | 确认无误后提交代码 |
| **触点** | 审查通过 → Git 提交 |
| **行为** | ① 看到「审查通过」提示<br>② 复制审查报告（可选）<br>③ 提交代码到仓库 |
| **想法** | "现在提交放心了"<br>"审查报告可以附在 PR 里" |
| **情绪** | 📈 自信 → 📈 满意 |

---

### 代码审查旅程总结

```
┌─────────────────────────────────────────────────────────────────────────┐
│  核心价值：在提交前发现 80% 的基础问题，减少线上故障                      │
│  关键指标：平均每次审查节省 15 分钟人工审查时间                           │
│  使用频率：每次代码提交前（高频场景）                                     │
└─────────────────────────────────────────────────────────────────────────┘
```

## 设计启示

### 基于旅程的设计原则

| 原则 | 说明 | 来源 |
|------|------|------|
| **首次体验即震撼** | 多专家并行响应是核心卖点，必须流畅 | Day 1 高点 |
| **冲突可视化** | 意见冲突时清晰对比，让用户决策 | Day 2 体验 |
| **Dashboard 驱动留存** | ROI 展示是付费转化关键 | Day 15-21 |
| **代码审查高频** | 优化审查流程，降低使用门槛 | 专项旅程 |
| **渐进式定制** | 从预配置到自定义，降低门槛 | Day 22-30 |

### 功能优先级（基于旅程）

| 优先级 | 功能 | 支撑旅程 |
|--------|------|---------|
| P0 | 多专家并行发言 | Day 1 震撼时刻 |
| P0 | 冲突对比视图 | Day 2 决策时刻 |
| P0 | 代码审查流程 | 每次提交 |
| P1 | Dashboard | Day 15 付费转化 |
| P1 | 追问和上下文 | 日常深度使用 |
| P2 | 专家市场 | Day 22 定制需求 |
