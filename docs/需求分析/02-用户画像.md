## 一、访谈核心洞察

### 产品定位重大调整

基于用户访谈，产品定位从「团队协作工具」调整为「**个人开发者效率工具**」：

```
┌─────────────────────────────────────────────────────────────────┐
│  调整前 → 调整后                                                │
├─────────────────────────────────────────────────────────────────┤
│  ❌ 多用户协作        →  ✅ 个人独立使用                        │
│  ❌ 网页/云端应用    →  ✅ PC桌面应用（本地安装）              │
│  ❌ B2B团队场景      →  ✅ 个人开发者场景                      │
│  ❌ 王经理角色       →  ✅ 删除（无团队需求）                  │
└─────────────────────────────────────────────────────────────────┘
```

### 关键发现

| 发现 | 影响 |
|------|------|
| 用户是**程序员**，不是产品经理 | 功能聚焦软件开发全流程 |
| 要**PC桌面应用**，科技风界面 | 技术栈转向 Electron/Tauri |
| **多专家并行发言**是核心 | 实现并行 LLM 调用架构 |
| **冲突时列出观点**让用户决策 | 不做自动仲裁，保持透明 |
| 每个人**独立使用**，配置可Git共享 | 无服务端，纯本地存储 |
| 需要**Dashboard**看效率/成本/进度 | 新增数据可视化模块 |

## 三、主要用户画像

### 画像：全栈开发者

```
┌─────────────────────────────────────────────────────────────────┐
│  📋 基础信息                                                     │
├─────────────────────────────────────────────────────────────────┤
│  姓名: 开发者（通用代表）                                        │
│  年龄: 25-35 岁                                                  │
│  职业: 全栈开发工程师 / 独立开发者                              │
│  团队规模: 7-9人（各自独立使用工具）                            │
│  技术栈: React/Vue + Node/Python + 各种新技术                   │
│  常用工具: Claude Code, VS Code, Git, Docker                    │
└─────────────────────────────────────────────────────────────────┘
```

#### demographic 特征

- **性别**: 男（占大多数）
- **年龄段**: 25-35 岁
- **收入水平**: 20-40K/月
- **工作地点**: 一线城市 / 远程办公
- **设备**: MacBook Pro（主力），外接大屏显示器
- **系统**: macOS / Linux（Windows 较少）

#### 心理特征

| 维度 | 描述 |
|------|------|
| **价值观** | 追求极致效率，相信好工具能 10x 生产力 |
| **生活方式** | 长时间编码，讨厌重复劳动，喜欢自动化 |
| **技术态度** | 愿意尝鲜，但要求工具专业、可定制 |
| **决策因素** | 效率提升 > 学习成本 > 价格 |

#### 行为特征

- 每天使用 Claude Code 辅助开发
- 经常需要切换不同角色视角（产品、架构、UI、前后端、QA、运维）
- 习惯用 Git 管理代码，用 Markdown 写文档
- 喜欢终端/TUI 界面，但也接受精美的 GUI
- 会为了好工具付费（Copilot、Cursor 等）

#### 目标与动机

**目标**:
1. 减少角色切换的上下文成本（不再手动改提示词）
2. 获得多专家并行反馈，提升决策质量
3. 自动化重复工作（代码审查、文档更新）
4. 追踪效率和成本，证明工具 ROI

**动机**:
- ⚡ **效率至上**：同一时间内获得更多专业意见
- 🎯 **质量追求**：多角度审视方案，减少盲点
- 💰 **成本意识**：时间 = 金钱，愿意为好工具付费
- 🔧 **掌控感**：自定义专家团队，打造个人工作流

#### 痛点与挫折

| 痛点 | 具体表现 | 严重程度 |
|------|---------|---------|
| **手动切换角色** | 每次都要输入"作为产品经理...""作为架构师..." | ⭐⭐⭐⭐⭐ |
| **没有界面** | CLI 记不住命令，看不到历史记录 | ⭐⭐⭐⭐⭐ |
| **不能并行** | 只能逐个问，效率低下 | ⭐⭐⭐⭐⭐ |
| **上下文丢失** | 换角色后需要重新解释背景 | ⭐⭐⭐⭐ |
| **无法追踪** | 不知道花了多少 token，省了多少钱 | ⭐⭐⭐⭐ |
| **冲突处理** | 不同角色观点冲突时，需要自己整合 | ⭐⭐⭐ |

## 四、核心使用场景

### 场景 1: 技术方案设计（高频）

```
┌─────────────────────────────────────────────────────────────────┐
│  场景: 设计一个新的系统模块                                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  触发: 收到一个新功能需求                                        │
│                                                                 │
│  用户操作:                                                      │
│  1. 打开应用，进入「技术方案设计」群组（固定工作组）             │
│  2. 群组专家：产品经理、架构师、前端、后端、安全、性能          │
│  3. 输入："设计一个用户积分系统，支持积分获取和消耗"           │
│  4. 各专家并行发言：                                            │
│     • 产品经理：业务场景分析、用户故事                          │
│     • 架构师：系统架构、数据模型、接口设计                      │
│     • 前端：页面结构、交互流程                                  │
│     • 后端：数据库设计、API 设计                                │
│     • 安全：潜在安全风险、防护措施                              │
│     • 性能：预估 QPS、缓存策略                                  │
│  5. 用户发现架构师建议用 Redis，性能专家建议用本地缓存          │
│  6. 系统列出两种方案的优缺点对比                                 │
│  7. 用户决策：选择 Redis 方案（可持久化）                       │
│  8. 追问细节：@架构师 "Redis 的数据结构怎么设计？"              │
│  9. 继续多轮讨论，完善方案                                       │
│  10. 导出最终方案为 Markdown，存到项目文档                      │
│                                                                 │
│  期望结果: 30 分钟完成原本需要 2 天的方案设计                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 场景 2: 代码审查（高频）

```
┌─────────────────────────────────────────────────────────────────┐
│  场景: 提交代码前自动审查                                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  触发: 完成一段代码编写，准备提交                                │
│                                                                 │
│  用户操作:                                                      │
│  1. 选择「代码审查」群组（或右键代码选择"开始审查"）            │
│  2. 群组专家：代码规范专家、性能专家、安全专家                  │
│  3. 粘贴代码或自动读取当前文件                                   │
│  4. 各专家并行审查：                                            │
│     • 代码规范：命名、格式、注释                                │
│     • 性能专家：时间复杂度、空间复杂度、N+1 问题                │
│     • 安全专家：SQL 注入、XSS、敏感信息泄露                     │
│  5. 发现问题：性能专家指出缺少索引，安全专家提示输入未校验      │
│  6. 用户修复问题，重新审查                                       │
│  7. 审查通过，提交代码                                          │
│                                                                 │
│  期望结果: 在提交前发现 80% 的基础问题                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 七、设计原则（基于访谈）

| 原则 | 说明 |
|------|------|
| **科技风界面** | 深色主题、代码高亮、精致表格 |
| **类微信交互** | 左侧联系人，右侧聊天，降低学习成本 |
| **开箱即用** | 预配置专家，不用写提示词 |
| **并行效率** | 多专家同时响应，最大化效率 |
| **透明决策** | 冲突时展示各方观点，不替用户做决定 |
| **本地优先** | 纯本地运行，数据不出机器 |
